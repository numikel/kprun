# kprun

Local secrets injector for developers and AI agent workflows. Secrets live in a KeePass `.kdbx` vault (compatible with [KeePassXC](https://keepassxc.org/)), unlock via OS keychain or prompt, and inject into a **single child process** as environment variables — scoped, not session-wide.

## Install

**Linux / macOS**

```bash
curl -fsSL https://raw.githubusercontent.com/numikel/kprun/refs/heads/master/scripts/install.sh | sh
```

**Windows (PowerShell)**

```powershell
irm https://raw.githubusercontent.com/numikel/kprun/refs/heads/master/scripts/install.ps1 | iex
```

Open a new terminal, then verify:

```bash
kprun --version
```

Rust developers can also build from source: `cargo install --path crates/kprun`.

## Quick start

```bash
# 1. Create vault (prompts for master password; stores it in OS keychain by default)
kprun init

# 2. Add secrets (entry title = service name; custom attributes = env var names)
kprun set github GITHUB_TOKEN=ghp_xxx

# 3. Inject into a child process
kprun run github -- npx -y @modelcontextprotocol/server-github
```

On Windows, the same flow applies after `install.ps1`:

```powershell
kprun init
kprun set github GITHUB_TOKEN=ghp_xxx
kprun run github -- npx -y @modelcontextprotocol/server-github
```

### Attach an existing KeePassXC database

```bash
kprun init --db /path/to/existing.kdbx
```

This verifies unlock credentials and optionally stores the master password in the OS keychain. It does **not** recreate the database.

## Environment variables

| Variable | Default | Description |
|----------|---------|-------------|
| `KPRUN_DB` | `~/.kprun/secrets.kdbx` | Path to the KeePass database |
| `KPRUN_KEYFILE` | — | Path to a cryptographic key file (second factor) |
| `KPRUN_LOG` | `~/.kprun/access.log` | Audit log path (JSON lines, key names only) |

## CLI reference

```
kprun init   [--db PATH] [--no-store] [--keyfile PATH]
kprun run    <entry> [entry2 ...] -- <command> [args...]
kprun list   [--json]
kprun get    <entry> [--keys] [--reveal]
kprun set    <entry> KEY=val [KEY2=val2 ...]
kprun unset  <entry> KEY [KEY2 ...]
kprun delete <entry>
kprun export [--format json|dotenv] [--stdout] [--reveal]
kprun import <file> [--merge]
kprun doctor [--mcp <entry>]
```

- `get` and `export` show key names by default; use `--reveal` only when you need values.
- `run` inherits stdin/stdout/stderr to the child and prints nothing on stdout itself (MCP-safe).

## MCP integration

MCP clients spawn a command and talk over the child's stdio. Use `kprun run` as a transparent wrapper.

**Linux / macOS** (with `kprun` on `PATH`):

```json
{
  "mcpServers": {
    "github": {
      "command": "kprun",
      "args": ["run", "github", "--", "npx", "-y", "@modelcontextprotocol/server-github"]
    }
  }
}
```

**Windows** (prefer full path to `kprun.exe` — some MCP clients do not resolve `PATH` reliably):

```json
{
  "mcpServers": {
    "github": {
      "command": "C:\\Users\\you\\AppData\\Local\\kprun\\bin\\kprun.exe",
      "args": ["run", "github", "--", "npx", "-y", "@modelcontextprotocol/server-github"]
    }
  }
}
```

Generate a ready-to-paste fragment for your install:

```bash
kprun doctor --mcp github
```

After editing MCP config, check `git diff` — some tools may write secrets back to disk.

## Automation and cron (`KPRUN_KEYFILE`)

When no interactive session is available (cron, Task Scheduler), the OS keychain may be unavailable. Use a **keyfile** as a second factor:

```bash
kprun init --keyfile ~/.kprun/master.key
export KPRUN_KEYFILE=~/.kprun/master.key
kprun run myservice -- /path/to/script.sh
```

The keyfile is a **cryptographic key file** generated by kprun — not a plaintext password file. Restrict file permissions (e.g. `chmod 600` on Unix; ACL for your user only on Windows).

For scheduled jobs, set `KPRUN_DB` and `KPRUN_KEYFILE` in the job environment and use `--no-store` at init if you do not want the master password in the keychain.

## Security notes

- Use a dedicated **dev-secrets** vault, not your personal password manager.
- Secrets exist in the child process environment after injection (same model as other secret runners).
- Audit log records entry names and injected key names — **never values**.
- Do not use `setx` or global shell profiles for API keys.
- Pass only the entries a command needs: `kprun run openai -- python script.py`, not every secret at once.

## KeePassXC compatibility test

Developers can verify KeePassXC-created databases locally:

1. Create `tests/fixtures/keepassxc.kdbx` in KeePassXC (gitignored).
2. Entry title: `fixture`; custom attribute: `FIXTURE_KEY`.
3. Run: `KPRUN_KEEPASSXC_FIXTURE=1 KPRUN_TEST_MASTER='...' cargo test reads_keepassxc_fixture -- --ignored`

## Polski (skrót)

`kprun` trzyma sekrety w bazie KeePass (`.kdbx`, zgodnej z KeePassXC), odblokowuje ją przez keychain systemowy lub hasło, i wstrzykuje zmienne środowiskowe tylko do jednego procesu potomnego.

```bash
curl -fsSL https://raw.githubusercontent.com/numikel/kprun/refs/heads/master/scripts/install.sh | sh
kprun init
kprun set github GITHUB_TOKEN=ghp_xxx
kprun run github -- npx -y @modelcontextprotocol/server-github
```

Zmienne: `KPRUN_DB`, `KPRUN_KEYFILE`, `KPRUN_LOG`. Do crona bez sesji użytkownika ustaw `KPRUN_KEYFILE` i ogranicz uprawnienia do pliku klucza.
