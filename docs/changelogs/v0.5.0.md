# Changelog v0.5.0

All notable changes to kprun are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [0.5.0] - 2026-07-12

### Added

- `kprun init --quick` — non-interactive onboarding that generates a master password, creates
  a password-only vault, stores the password in the OS keychain, and prints it once for the user.
  Supports `--force` flag to overwrite an existing vault (with confirmation if running in a TTY).
- `kprun reveal-master` — retrieve and display the stored master password from the OS keychain.
  Logs access to the audit trail (never passwords). Supports `--db` flag to target a vault at a
  non-standard path.
- `kprun deinit --delete-vault` — remove both the vault file and its associated keychain entry.
  Supports `--db` flag for consistency with `init --quick` and `reveal-master`.

### Changed

- OS keychain account name derivation: now computes SHA-256 of the lexically-absolutized database
  path (via `std::path::absolute`) instead of `fs::canonicalize`. This ensures the account name
  remains stable even when the vault file does not exist yet, enabling consistent lookups across
  all three commands.

### Fixed

- `init --quick --force` now uses atomic vault creation (write to temporary sibling, then rename
  into place) ensuring that keychain storage always succeeds before committing the file; a failure
  never leaves the user without a usable vault.
- `deinit --delete-vault` now removes the vault file **before** deleting the keychain entry,
  preventing orphaned keychain passwords when file removal fails (e.g. vault is open in KeePassXC).
- `reveal-master` now maps backend keychain errors to actionable diagnostic messages rather than
  surfacing raw backend error strings.
- Keychain probe cleanup improved to best-effort remove probe entries even when the probe read fails.

### ⚠️ BREAKING CHANGES

The OS keychain account name scheme has changed. On platforms where the lexical and canonical
paths differ — **every Windows vault** (due to `\\?\` prefix) and **macOS vaults under `/tmp`** —
the account name changes, rendering previously stored master passwords inaccessible.

**Migration**: Users affected by this change must re-run `kprun init` (or manually re-store the
master password once) after upgrading. Password-only vaults created before this release, whose
vault files have since been deleted or moved, cannot be recovered via `reveal-master` — this is
a direct consequence of the migration.
